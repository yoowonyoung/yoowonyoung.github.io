# 의도를 분명히 밝혀라
- 의도가 분명한 이름은 정말로 중요하다
- 변수나 함수 그리코 클래스 이름은 다음과 같은 굵직한 질문에 모두 대답이 가능해야 한다. 주석이 필요하다면 의도를 드러내지 못했다는 말이다
    * 존재 이유는?
    * 수행 기능은?
    * 사용 방법은?

```java
int d; //경과 시간(단위: 날짜)

int elapsedTimeInDays;
int daysSinceCreation;
int daysSinceModification;
```

- 이름 ```d```는 아무 의미도 드러나지 않는다. 경과 시간이나 날짜라는 느낌이 안든다. 측정하려는 값과 단위를 표현한 이름이 훨씬 가독성이 좋다
- 의도가 드러나는 이름을 사용하면 코드 이해와 변경이 쉬워진다

```java
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<int[]>();

    for(int[] x : theList) {
        if(x[0] == 4)
            list1.add(x);
    }

    return list1;
}
```

- 위의 코드가 하는일은 짐작하기 어렵다. 복잡한 문장은 하나도 없고, 변수도 3개, 상수는 2개 뿐이지만 문제는 코드의 단순성이 아닌 함축성이다. 코드의 맥락이 코드 자체에 명시적으로 드러나지 않기 떄문이다.
- 위의 코드를 이해하기 위해서는 다음과 같은 정보를 알고 있어야 한다
    * theList에 무엇이 들어 있는지
    * theList에서 0번째 값이 왜 중요한지
    * 값 4는 무슨 의미인지
    * 함수가 반환하는 list1을 어떻게 사용 하는지

- 위의 코드에서는 이러한 정보가 드러나지 않지만, 정보 제공은 충분히 가능 하다
- 위의 코드가 지뢰 찾기 게임을 만든다고 한다면 theList가 게임판이라는 사실을 알 수 있다. 또 게임판에서 각 칸은 단순 배열이고, 각 배열의 0번째값은 칸 상태를 뜻하며, 값 4는 깃발이 꽂힌 상태이다. 이런 개념을 이용해 각 개념에 이름만 붙여도 코드는 훨씬 나아진다

```java
public List<int[]> getFlaggedCells() {
    List<int[]> flaggedCells = new ArrayList<int[]>();
    
    for(int[] cell : gameBoard) {
        if(cell[STATUS_VALUE] == FLAGGED)
            flaggedCells.add(cell);
    }

    return flaggedCells;
}
```

- 한걸음 더 나아가, int 배열을 사용하는 대신, 칸을 간단한 클래스로 만들어도 된다. Cell 클래스에 isFlagged라는 좀 더 명시적 함수를 사용해 FLAGGED 상수를 감춰도 좋다

```java
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<Cell>();
    
    for(Cell cell : gameBoard) {
        if(cell.isFlagged())
            flaggedCells.add(cell);
    }

    return flaggedCells;
}
```


# 그릇된 정보를 피하라
- 프로그래머는 코드에 그릇된 단서를 남겨서는 안된다. 그릇된 단서는 코드의 의미를 흐리며, 나름대로 널리 쓰이는 의미가 있는 단어를 다른 의미로 사용해도 안된다
- 예를 들어, 여러 계정을 그룹으로 묶을때 실제 List가 아니면 accountList라고 명명하면 안된다. 프로그래머에게 List란 특수한 의미이기 때문이다
- 서로 흡사한 이름을 사용하지 않도록 주의해야 한다
- 유사한 개념은 유사한 표기법을 사용하도록 한다

# 의미 있게 구분하라
- 컴파일러나 인터프리터만 통과 하려는 생각으로 코드를 구현하면 안된다. 컴파일러를 통과하더라도 연속된 숫자를 멋붙이거나 불용어를 추가하는 방식은 적절하지 못하다. 이름이 달라지면 의미도 달라져야 한다
- a나 the같은 접두어를 사용하지 말라는 의미가 아니다. 의미가 분명히 다르다면 사용해도 무방하다
- 불용어는 중복이며, 변수 이름에 variable이라는 단어는 단연코 금물이다. 표 이름에 table이라는 단어도 마찬가지 이다

# 발음하기 쉬운 이름을 사용하라
- 발음하기 어려운 이름은 토론하기도 어렵다

# 검색하기 쉬운 이름을 사용하라
- 문자 하나를 사용하는 이름과 상수는 텍스트 코드에서 눈에 쉽게 띄지 않으며 검색하기도 어렵다
- 개인적으로 간단한 메서드에서 로컬 변수만 한 문자를 사용한다. 이름 길이는 범위의 크기에 비례해야 하며, 변수나 상수를 코드 여러곳에사 사용한다면 검색하기 쉬운 이름이 바람직 하다

# 인코딩을 피하라
- 굳이 부담을 더하지 않아도 이름에 인코딩할 정보는 아주 많은데, 유형이나 범위 정보까지 인코딩에 넣으면 그만큼 이름을 해독하기 어려워 진다
- 더이상 헝가리안 표기법, 멤버변수 접두어 같은걸 사용하지 말아야 한다
- 인터페이스와 구현 클래스와 같이 인코딩이 필요한 경우가 있기는 한데, 차라리 구현 클래스 이름을 ~Impl 정도로 해주는게 낫다

# 자신의 기억력을 자랑하지 마라
- 독자가 코드를 읽으면서 변수 이름을 자신이 아는 이름으로 변환해야 한다면, 그 변수 이름은 바람직하지 못한것이다
- 문자를 하나만 사용하는 변수 이름은 문제가 있다
    * 반복문에서 루프의 범위가 아주 작고, 다른 이름과 충돌하지 않는 경우에만 i,j,k는 괜찮다
    * l은 절대 안된다

- 전문가 프로그래머는 명료함이 최고라는 사실을 이해하고 있는데, 이를 이용해 남들이 이해하는 코드를 만들어 내야 한다

# 클래스 이름
- 클래스 이름과 객체 이름은 명사나 명사 구가 적합하다
- Manager, Processor, Data, Info 같은 단어는 피하고 동사는 사용하지 않는다

# 메서드 이름
- 메서드 이름은 동사나 동사구가 적합하다
- 접군자, 변경자, 조건자는 표준에 따라 get, set, is를 앞에 붙인다
- 생성자를 중복 정의할때에는 정적 팩토리 메서드를 사용한다. 메서드는 인수를 설명하는 이름을 사용한다

# 기발한 이름은 피하라
- 이름이 너무 기발하면 저자와 유머감각이 비슷하거나, 그 농담을 기억 하는 동안만 이름을 기억 할것이다
- 재미난 이름보다는 명료한 이름을 선택해야한다

# 한 개념에 한 단어를 사용하라
- 추상적인 개념 하나에 단어 하나를 선택해 이를 고수한다. 똑같은 메서드를 클래스마다 fetch, retrieve, get으로 제각각으로 표현하면 혼란스럽다

# 말장난을 하지마라
- 한 단어를 두가지 목적으로 사용하지마라. 다른 개념에 같은 단어를 사용한다면 그것은 말장난에 불과하다
- 하지만 때로는 같은 맥락이 아닌데도 일관성을 고려하여 같은 이름을 사용하려는 경우가 있는데, 이는 맥락이 다르므로 다른 이름을 사용해야 한다

# 해법 영역에서 가져온 이름을 사용하라
- 코드를 읽을 사람도 프로그래머이기 때문에, 전산 용어, 알고리즘 이름, 패턴 이름, 수학 용어등을 사용해도 괜찮다
- 모든 이름을 문제 영역에서 가져오는 정책은 현명하지 못하다
- 기술 개념에는 기술 이름이 가장 적합한 선택이다

# 문제 영역에서 가져온 이름을 사용하라
- 적절한 프로그래머 용어가 없을 경우에만 문제 영역에서 이름을 가져 오도록 한다
- 우수한 프로그래머와 설계자라면 해법 영역과 문제 영역을 구분할줄 알아야 한다

# 의미 있는 맥락을 추가하라
- 스스로 의미가 분명한 이름이 없지 않은데, 대다수의 이름은 그렇지 못하다
- 그러므로 클래스, 함수, 이름 공간에 넣어 맥락을 부여해야 한다. 모든 방법이 실패 할때만 접두어를 사용한다

# 불필요한 맥락을 없애라
- 고급 휘발유 충전소(Gas Station Deluxe) 어플리케이션을 만든다고 해서 모든 클래스 이름을 GSD로 시작 하겟다는 생각은 전혀 바람직 하지 못하다
- 일반적으로는 짧은 이름이 긴 이름보다 좋지만, 의미가 분명한 경우에 한해서 이다
- 이름에 불필요한 맥락을 추가하지 않도록 주의해야 한다
